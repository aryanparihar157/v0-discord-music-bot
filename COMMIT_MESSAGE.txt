Fix: Enable Songs to Play and Not Just Queue

## Problem
Songs were being queued but not actually playing. Root cause was that the slash command handler was not connected to the music playback system - it was only returning Discord messages without triggering actual playback.

## Changes Made

### 1. Fixed Slash Command Handler (bot/handlers/slashCommandHandler.ts)
- Rewrote all command handlers to use actual music state management
- Added imports for getOrCreateMusicState, addToQueue, playNextSong
- Implemented real playback logic in /play command
- Added voice connection subscription to the audio player
- Implemented working pause/resume/skip/stop/volume/nowplaying controls
- Added comprehensive console logging with [v0] prefix for debugging

### 2. Enhanced Play Command (bot/commands/play.ts)
- Added voice channel subscription: connection.subscribe(musicState.player)
- This ensures audio is actually sent to Discord voice channel
- Added detailed logging at each step

### 3. Improved Music State (bot/utils/musicState.ts)
- Enhanced playNextSong() with detailed debugging logs
- Better error handling with fallback to next song on failure
- Clear logging of song transitions

### 4. Better Streaming (bot/utils/streaming.ts)
- Improved YouTube streaming with detailed error messages
- Added logging for stream creation and status
- Better error recovery

### 5. Fixed Dependencies
- Changed play-dl from ^1.10.5 (doesn't exist) to ^1.9.7 (latest available)
- Added .eslintrc.json for ESLint support

## How Songs Play Now
1. User types /play [song name]
2. Handler searches for song using musicSearch
3. Song is added to queue via addToQueue()
4. If nothing is playing, playNextSong() is called immediately
5. Audio stream is created and played through voice connection
6. Bot streams audio to Discord voice channel

## Testing
All console logs use [v0] prefix for easy debugging. Check logs when testing:
- [v0] Added "..." to queue
- [v0] Starting playback of: ...
- [v0] Creating audio resource
- [v0] ✓ Now playing: ...
